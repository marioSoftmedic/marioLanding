# El error de base de datos que destruye tus convenios de salud B2B

Cuando una clínica mediana firma un convenio con un laboratorio de referencia o una red de salud, el acuerdo comercial se celebra con bombos y platillos. 

Se definen tarifas, SLAs y flujos de derivación. Pero casi nadie invita al arquitecto de software a esa mesa.

El resultado es predecible: el convenio se implementa en producción usando parches en el software existente. Y ahí es donde empieza la pesadilla del cumplimiento regulatorio y la seguridad de datos.

En Chile, con la Ley 21.668 de interoperabilidad y la Ley 21.719 de datos clínicos, delegar la seguridad del convenio a "filtros lógicos" básicos en tu backend es un suicidio corporativo.

## La trampa de los roles globales planos

La mayoría del software clínico legacy (y muchas startups apresuradas) resuelven la autorización B2B añadiendo un rol a su tabla de usuarios:

```sql
-- La forma incorrecta y peligrosa
ALTER TYPE "UserRole" ADD VALUE 'DPO';
ALTER TYPE "UserRole" ADD VALUE 'CLINIC_ADMIN';
```

A primera vista parece lógico. Le asignas el rol `CLINIC_ADMIN` al supervisor de la clínica socia para que pueda ver los resultados de sus pacientes. Luego, en tu backend, haces algo como:

```typescript
// Pseudocódigo que causa fugas de datos (Tenant Leaks)
if (user.role === 'CLINIC_ADMIN') {
  return db.patients.findMany({ where: { originClinicId: user.clinicId } });
}
```

¿Cuál es el problema? Has convertido una relación dinámica, contractual y sujeta a auditorías (el convenio) en un atributo estático y global del usuario. 

Si el administrador de la clínica A trabaja también los sábados en la clínica B, el modelo se rompe. Si la clínica A revoca el convenio, tienes que ir a buscar manualmente cada usuario para cambiar su rol en lugar de suspender la entidad comercial. Y lo peor: estás a una sola consulta SQL mal filtrada de exponer exámenes de una institución rival.

## La solución: Modelar el convenio como infraestructura

Cuando estructuramos la capa B2B de interoperabilidad en Examya y Fhirex, decidimos que la pertenencia a una institución no es un rol. Es un contrato explícito con políticas de seguridad a nivel de fila (Row-Level Security) en la base de datos.

Incoherente a roles globales, creamos una estructura basada en membresías institucionales (`InstitutionMembership`) vinculadas a convenios específicos (`ConvenioInstitucional`):

```typescript
// Esquema conceptual en Drizzle ORM
export const institutionMembership = pgTable("institution_memberships", {
  id: uuid("id").primaryKey().defaultRandom(),
  userId: uuid("user_id").references(() => users.id).notNull(),
  institutionId: uuid("institution_id").references(() => institutions.id).notNull(),
  role: text("role", { enum: ["admin", "validator", "viewer"] }).notNull(),
  status: text("status", { enum: ["active", "suspended", "revoked"] }).default("active"),
  createdAt: timestamp("created_at").defaultNow()
});
```

Este cambio de paradigma cambia todo:

1. **Aislamiento Multi-Tenant Real:** Las consultas a nivel de API no dependen de si el programador se acordó de filtrar por `clinicId`. La base de datos, usando políticas RLS automáticas basadas en la sesión del token JWT, solo devuelve los registros vinculados a la membresía activa del usuario.
2. **Ciclo de Vida del Convenio:** Si un convenio comercial se congela por impago o término de contrato, cambias el estado de la institución a `suspended`. Automáticamente, todas las membresías vinculadas pierden el acceso a la API, sin tener que tocar las cuentas individuales de los médicos o secretarias.
3. **Pistas de Auditoría para Cumplimiento:** Cada evento de lectura o escritura de datos clínicos (como las firmas de resultados) no se registra solo bajo el ID del usuario, sino que incluye el ID del convenio institucional que autorizó dicho acceso. Trazabilidad absoluta para auditorías de la Ley 21.719.

## El Zoom Out

Si estás liderando una red de salud o un laboratorio y estás buscando interconectar tu operation con terceros mediante APIs, no te limites a mirar las pantallas de la interfaz o la velocidad de las respuestas.

Abre el esquema de base de datos. Pregunta cómo se maneja el multi-tenancy y cómo se segmentan los datos cuando expongas endpoints. Si la respuesta es "lo filtramos en el código del controlador", da un paso atrás. En la era de la interoperabilidad obligatoria, tu base de datos debe ser el primer y más estricto firewall de cumplimiento.
