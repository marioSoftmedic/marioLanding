## 1. El Hook

Generar un video con un agente no demuestra que ese mismo archivo quedó listo para publicación. Entre la exportación y el handoff humano pueden cambiar el contenido, las dependencias o el puntero a la generación vigente.

La pregunta útil no es solo “¿existe el MP4?”, sino “¿qué evidencia liga este MP4 con la generación que está vigente ahora?”.

## 2. Lo que nadie cuenta

Un hash aislado tampoco resuelve el problema. Puede describir un archivo antiguo mientras el puntero actual ya referencia otra generación.

Por eso el pipeline editorial separa cuatro objetos: una generación inmutable, un puntero actual, un receipt consumidor ligado a ese puntero y un bundle visible que el operador puede revisar. Es un patrón de publicación de activos, no una afirmación sobre productos clínicos.

## 3. Los datos reales

La evidencia está en tres fuentes canónicas del pipeline:

- `export-reel.mjs` instala cada ejecución completa bajo `.reel-generations/<basename>/<runId>/` y cambia de forma atómica el puntero `<basename>.current` (`C-PIPE-001`).
- `verify-reel.mjs` escribe un receipt consumidor que liga el `runId`, el puntero, los artefactos, las pruebas visuales y los hashes de dependencias (`C-PIPE-002`).
- `visible-bundle.mjs` crea un directorio real con `PUBLISH-ME.mp4`, archivos de soporte, pruebas visuales y `handoff-verification.json`, y luego comprueba que siga ligado al receipt actual (`C-PIPE-003`).

Las rutas fuente están bajo `~/.agents/skills/examya-social-editorial/assets/reel-templates/`. No hay un commit externo usado como sustituto de esta evidencia.

## 4. Lo que construiste

El flujo demostrable tiene cuatro etapas:

1. El exportador produce HTML, WEBM, MP4, verificación y una prueba visible por beat dentro de una generación inmutable.
2. El puntero `.current` identifica la generación vigente.
3. El verificador consumidor vuelve a calcular la evidencia y escribe un receipt estable ligado a esa generación.
4. El creador del bundle visible copia solo archivos regulares de esa generación y deja un `PUBLISH-ME.mp4` comprobable para el operador.

Si cambia el puntero o cualquiera de los hashes ligados, los checks dejan de pasar. Eso bloquea el handoff editorial; no ejecuta ni demuestra un deploy.

## 5. El gotcha

El gotcha es temporal: un receipt válido puede volverse obsoleto sin que el archivo que describe cambie.

Por eso `check-receipt` no se limita a leer JSON. Contrasta el receipt con el puntero vigente y con los hashes actuales. El check del bundle visible repite esa igualdad antes del handoff. La validez pertenece a una ejecución concreta, no al basename para siempre.

## 6. Zoom out

Este patrón prueba algo acotado: que un activo visible del pipeline editorial coincide con una generación verificada y con su receipt consumidor actual (`C-PIPE-004`). Reduce el riesgo de entregar una grabación obsoleta o alterada dentro de este flujo.

No prueba integridad de órdenes médicas, resultados, cotizaciones, datos de pacientes ni comportamiento clínico, ni ofrece garantías de privacidad. Tampoco prueba que el activo fue desplegado o publicado. Extender esas garantías exigiría evidencia específica del producto y de cada superficie.

## 7. Lo que viene

El siguiente paso editorial es ejecutar el checker formal, sincronizar el contenido corregido con el repositorio del sitio y validar build y despliegue antes de hablar de publicación.

El siguiente paso técnico es mantener estos checks como gate de cada nuevo Reel y registrar cualquier cambio de dependencia como una nueva generación, no como una validación heredada.
