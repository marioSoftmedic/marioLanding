## 1. The Hook

Generating a video with an agent does not prove that the same file is ready for publication. Between export and human handoff, the content, dependencies, or pointer to the current generation may change.

The useful question is not only “does the MP4 exist?” but “what evidence binds this MP4 to the generation that is current now?”

## 2. What People Leave Out

An isolated hash does not solve the problem either. It may describe an old file while the current pointer already references a different generation.

That is why the editorial pipeline separates four objects: an immutable generation, a current pointer, a consumer receipt bound to that pointer, and a visible bundle the operator can review. This is a publishing pattern for media assets, not a claim about clinical products.

## 3. The Actual Evidence

The evidence lives in three canonical pipeline sources:

- `export-reel.mjs` installs each complete run under `.reel-generations/<basename>/<runId>/` and atomically switches the `<basename>.current` pointer (`C-PIPE-001`).
- `verify-reel.mjs` writes a consumer receipt that binds the `runId`, pointer, artifacts, visual proofs, and dependency hashes (`C-PIPE-002`).
- `visible-bundle.mjs` creates a real directory with `PUBLISH-ME.mp4`, support files, visual proofs, and `handoff-verification.json`, then checks that it remains bound to the current receipt (`C-PIPE-003`).

The source paths live under `~/.agents/skills/examya-social-editorial/assets/reel-templates/`. No external commit is used as a substitute for this evidence.

## 4. What the Pipeline Does

The demonstrable flow has four stages:

1. The exporter produces HTML, WEBM, MP4, verification, and one visible proof per beat inside an immutable generation.
2. The `.current` pointer identifies the active generation.
3. The consumer verifier recalculates the evidence and writes a stable receipt bound to that generation.
4. The visible-bundle creator copies only regular files from that generation and leaves a verifiable `PUBLISH-ME.mp4` for the operator.

If the pointer or any bound hash changes, the checks stop passing. That blocks the editorial handoff; it neither runs nor proves a deployment.

## 5. The Gotcha

The gotcha is temporal: a valid receipt can become stale even when the file it describes has not changed.

That is why `check-receipt` does more than read JSON. It compares the receipt with the current pointer and current hashes. The visible-bundle check repeats that equality before handoff. Validity belongs to one specific run, not to the basename forever.

## 6. Zooming Out

This pattern proves one narrow property: a visible asset in the editorial pipeline matches a verified generation and its current consumer receipt (`C-PIPE-004`). It reduces the risk of handing off a stale or altered recording within this workflow.

It does not prove the integrity of medical orders, results, quotes, patient data, or clinical behavior, and it does not establish privacy guarantees. It also does not prove that the asset was deployed or published. Extending those guarantees would require product-specific evidence for every surface.

## 7. What Comes Next

The next editorial step is to run the formal checker, synchronize the corrected content with the site repository, and validate the build and deployment before claiming publication.

The next technical step is to keep these checks as the gate for every new Reel and treat any dependency change as a new generation, not inherited validation.
