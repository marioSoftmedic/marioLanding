El debate entre modelos de lenguaje a menudo se centra en benchmarks y teoría. Pero en la práctica diaria de desarrollo, especialmente para español, la elección técnica real cambia completamente el panorama.

## Problema inicial: la limitación de español en GPT-4

Cuando comencé Examya, usaba GPT-4 como motor principal. El primer problema fue claro: respuestas en español con estructura anglosajona. Código con comentarios en inglés, variables descriptivas en español, y un choque constante de patrones.

GPT-4 sigue un patrímoma británico/estadounidense por defecto. Incluso con prompts explícitos en español, la salida mantén estructuras forzadas. Preguntaba por "función para validar RUT chileno" y obtenía código con docstrings en inglés, nombres de variables camelCase británico, y lógica optimizada para casos de uso inglés.

La solución no fue más prompts. Fue cambiar de motor.

**Patrones de código auténticos**:
```typescript
// Lo que GPT-4 genera (forzado)

  // British-style docstring
  // Validates Chilean RUT number format
  // Returns boolean indicating validity

// Lo que Claude genera (natural)

  // Valida formato de RUT chileno
```

**Flujo de pensamiento alineado**: cuando pregunto por "manejo de errores en WhatsApp API", Claude entiende inmediatamente el contexto de mensajes fallidos, retries, y patrones chilenos. GPT-4 requiere explicación del contexto específico.

## Costo real: no solo tokens

La diferencia no está en precio por token. Está en iteraciones:

**GPT-4**:
- Prompt: "Crea función para enviar mensajes WhatsApp"
- Primera versión: código con estructura de librería británica
- Segunda versión: ajuste para contexto chileno
- Tercera versión: corrección de errores de lógica
- 3 iteraciones, 4x el costo real

**Claude**:
- Prompt: "Función para enviar mensajes WhatsApp con manejo de errores"
- Primera versión: código listo con patrones chilenos correctos
- Segunda versión: solo ajustes menores de estilo
- 1.5 iteraciones, 40% menos costo real

## Caso real: OCR médico

En Examya procesamos órdenes médicas por WhatsApp. Un caso concreto:

**GPT-4**:
- Generador de prompts para OCR con términos médicos en inglés
- Requería 5 revisiones para entender que "hemograma" no es "blood count" en contexto chileno
- Tokens consumidos: 15,000 para lograr resultado usable

**Claude**:
- Prompts OCR con términos médicos chilenos desde el inicio
- Entendió automáticamente la diferencia entre "glucosa" y "azúcar en sangre"
- Tokens consumidos: 3,200 para resultado óptimo

## Experiencia práctica con agentes

La diferencia más notable es con agentes autónomos como Shuri en Examya:

**Claude**:
- Mensajes de WhatsApp con tono natural chileno
- Manejo de contexto cultural ("cuélgalo" vs "sube el archivo")
- Entiende frases coloquiales como "sácame una cotización"

**GPT-4**:
- Mensajes más formales, con estructura internacional
- Requiere prompts específicos para capturar localismo
- Genera respuestas genéricas sin conexión emocional

## Herramientas técnicas

La ventaja de Claude es profunda en herramientas técnicas:

**Consistencia en español**:
- Comentarios en código siempre en español
- Documentación API con términos técnicos correctos
- Mensajes de error en contexto chileno

**Manejo de ambigüedad**:
- "Cotización" puede precio o presupuesto, contexto lo decide
- "Laboratorio" puede institución o espacio físico, contexto lo resuelve
- "Orden médica" vs "requisición", término correcto según FONASA

## ¿Cuándo usar GPT-4?

GPT-4 tiene ventajas claras:

- Code review internacional: patrones globales
- Traducción real: inglés a español técnico
- Documentación para equipos multilingües
- Integración con sistemas heredados

Pero para desarrollo 100% en español, especialmente con dominio técnico específico, Claude es simplemente más eficiente.
