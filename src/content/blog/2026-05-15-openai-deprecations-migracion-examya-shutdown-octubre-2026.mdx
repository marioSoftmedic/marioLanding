# OpenAI Deprecations: Cómo migramos Examya antes del shutdown del 23 de octubre

El 22 de abril de 2026, llegó el email que todo equipo de IA teme: OpenAI anunció la deprecación masiva de modelos clave. Entre ellos, `text-embedding-3-small` - el backbone del sistema de embeddings de Examya - con shutdown programado para el 23 de octubre de 2026.

Seis meses de runway. Una decisión crítica que afecta directamente cómo nuestro agente médico busca en el catálogo FONASA de exámenes.

## El problema real: embeddings en producción

En Examya, el sistema de embeddings no es un "nice to have". Es el núcleo de la inteligencia médica. Cuando un paciente envía una orden médica por WhatsApp:

```
Paciente: "Quiero hacer un examen de sangre completo"
→ Shuri (agente médico) genera embedding vectorial
→ pgvector busca en catálogo FONASA semánticamente
→ Retorna: "Exámenes disponibles: Hemograma completo, Perfil lipídico, Función renal..."
```

`text-embedding-3-small` era el modelo que convertía texto médico a vectores numéricos. Su deprecación no es solo un cambio de API: es una migración de todo el pipeline vectorial.

## La matriz de migración: qué cambia, qué se mantiene

Tras analizar el uso real en production y los requirements de FONASA, nuestra matriz de migración:

| Modelo Actual | Problema | Reemplazo | Estado |
|---------------|---------|-----------|---------|
| `text-embedding-3-small` | Shutdown 23-oct-2026 | `text-embedding-3-large` | ✅ Migrado |
| `gpt-4-turbo-preview` | Costo + speed | `gpt-4o` | ✅ Migrado |
| `gpt-3.5-turbo` | Context window | `gpt-3.5-turbo-0125` | ✅ Migrado |

El impacto más grande fue en `text-embedding-3-small`: embeddings 5x más grandes (1536 vs 768 dimensions), pero con calidad semántica significativamente mejor para términos médicos complejos.

## Código: el nuevo MLESearchAgent

La migración no fue solo cambiar el modelo. Fue reconstruir el pipeline embeddings:

```typescript
// ANTES: text-embedding-3-small (768 dims)
const embedding = await openai.embeddings.create({
  model: "text-embedding-3-small",
  input: medicalQuery,
  dimensions: 768
});

// DESPUÉS: text-embedding-3-large (1536 dims)
const embedding = await openai.embeddings.create({
  model: "text-embedding-3-large", 
  input: medicalQuery,
  dimensions: 1536
});
```

Pero el verdadero cambio fue en la capa de búsqueda:

```typescript
// Nuevo índice pgvector con mayor precisión
CREATE INDEX fonasa_search_idx ON medical_items 
  USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);

// Nueva lógica de búsqueda semántica
async function searchFONASA(query: string): Promise<MedicalItem[]> {
  const queryEmbedding = await generateEmbedding(query);
  
  return await prisma.medicalItem.findMany({
    where: {
      embedding: {
        vectorCosineDistance: queryEmbedding,
        lte: 0.3 // Umbral más estricto con embeddings 1536d
      }
    },
    orderBy: {
      embedding: {
        vectorCosineDistance: queryEmbedding
      }
    },
    take: 10
  });
}
```

## Pruebas: de unit tests a adversarial testing

La migración no fue sin friction. Descubrimos que `text-embedding-3-large`:

1. **Mejora la precisión**: +15% en recuperación de términos médicos complejos
2. **Aumenta el costo**: +200% por llamada de embedding  
3. **Ralentiza el proceso**: +30ms por búsqueda

Nuestro protocolo de testing:

```bash
# Test semántico con casos médicos reales
npm run test:embeddings
# → 87% precisión vs 72% anterior

# Test de carga real  
npm run test:search-stress --concurrency=50
# → P95: 450ms vs 380ms anterior

# Test de costos mensuales simulados
npm run test:cost-projection
# → $450/mes vs $150/mes anterior
```

## Lo que aprendí: 3 lecciones críticas

### 1. Nunca confíes en "will be deprecated" timelines

OpenAI anunció el shutdown con 6 meses de anticipación. Eso suena mucho, pero en producción:

- Tiempo para testing: 4 semanas  
- Tiempo para despliegue: 2 semanas
- Tiempo para rollback: 1 semana
- **Buffer real disponible: 5 semanas**

### 2. La calidad tiene costo (y vale la pena)

El +200% en costo de embeddings se compensó con:

- Menos falsos positivos en búsqueda
- Pacientes reciben exámenes más relevantes  
- Menos soporte por órdenes incorrectas

### 3. La arquitectura debe ser agnóstica al modelo

El mayor aprendizaje fue que nuestro código debería haber sido:

```typescript
interface EmbeddingProvider {
  generateEmbedding(text: string): Promise<number[]>;
}

class OpenAIEmbeddingProvider implements EmbeddingProvider {
  async generateEmbedding(text: string): Promise<number[]> {
    // Modelo específico aquí
  }
}

// Donde sea usado:
class MLESearchAgent {
  constructor(private embeddingProvider: EmbeddingProvider) {}
  
  async search(query: string): Promise<MedicalItem[]> {
    const embedding = await this.embeddingProvider.generateEmbedding(query);
    // ... búsqueda vectorial
  }
}
```

## Lo que viene: embeddings multi-proveedor

```typescript
class MultiEmbeddingProvider implements EmbeddingProvider {

```
