Puerto Natales, la puerta de entrada a Torres del Paine, merece un portal digital a la altura de su belleza. En los últimos meses, construí natales.travel: un destination portal completamente independiente que busca posicionarse como la fuente definitiva para viajeros que exploran la Patagonia chilena.

## El problema a resolver

La experiencia de planear un viaje a Puerto Natales era fragmentada. Los viajeros saltaban entre múltiples fuentes: blogs desactualizados, listas en redes sociales, y contactos directos con proveedores. Faltaba un centralizador que combinara:

- Información actualizada de experiencias
- Sistema de reservas integrado  
- Guías de viaje confiables
- Verificación de proveedores locales
- Diseño premium que transmitiera la majestuosidad del destino

## Arquitectura del stack

### Backend robusto con Prisma + PostgreSQL

Aunque mi stack principal en Examya usa Supabase, para natales.travel elegí PostgreSQL tradicional con Prisma. La decisión fue estratégica:

```typescript
// schema.prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model Experience {
  id          String   @id @default(cuid())
  title       String
  description String
  price       Float
  providerId  String
  category    Category
  isActive    Boolean  @default(true)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

enum Category {
  ACCOMMODATION
  ACTIVITY  
  DINING
  TRANSPORT
}
```

La decisión de usar PostgreSQL local (no Docker) fue clave para evitar conflictos de puerto con mi entorno de desarrollo existente.

### Frontend: Next.js 16 con Tailwind v4

La evolución del frontend merece atención especial. Next.js 16 trajo App Router estable, y Tailwind v4 cambió completamente el enfoque de configuración:

```css
/* globals.css */
@theme {
  --color-primary: #0f172a;
  --color-accent: #0891b2;
  --font-display: 'Inter', system-ui, sans-serif;
}

body {
  font-family: var(--font-display);
}
```

La configuración CSS-based en lugar de JS config simplificó enormemente el setup y permitió un control más granular sobre las variables del sistema.

### i18n nativo con routing basado en URL

Para el SEO multilingüe, implementé routing de URL en lugar de un toggle client-side:

```typescript
// src/i18n/utils.ts
export function getLang(pathname: string): 'en' | 'es' {
  if (pathname.startsWith('/es/')) return 'es'
  return 'en'
}

export function getAlternatePath(pathname: string, lang: 'en' | 'es'): string {
  const segments = pathname.split('/').filter(Boolean)
  const slug = segments[segments.length - 1]
  return lang === 'es' ? `/${slug}` : `/en/${slug}`
}
```

Esto permite que Google indexe versiones independientes de cada página en ambos idiomas.

## Decisions de diseño clave

### Branding completamente independiente

Una decisión importante fue separar visualmente natales.travel de mis otros proyectos. El portal tiene:

- Identidad propia: premium y cinematográfico
- Inspiración: National Geographic + lujo accesible  
- Paleta de colores: oscuros con acentos turquesa
- Tipografía: Inter para equilibrio entre legibilidad y elegancia

El diseño refleja la majestuosidad de Torres del Paine mientras mantiene usabilidad para viajeros que buscan información rápida.

### Arquitectura escalable: de MVP a plataforma completa

Diseñé el sistema para evolucionar en 3 fases:

**Fase A (Actual)**: Portal editorial curado
- Contenido manualmente verificado
- Experiencias seleccionadas
- Guías de escritura original

**Fase B**: Sistema de reservas integrado
- Backend de bookings
- Panel para proveedores
- Sistema de availability

**Fase C**: Plataforma completa
- Proveedores auto-registrados
- Sistema de reviews
- API pública para terceros

## Lo que aprendí

### Prisma 6 vs Supabase: cuándo usar cada uno

Aunque Supabase es excelente para proyectos que necesitan Firebase-like features, PostgreSQL + Prisma puro brilla cuando:

- Se requiere control total sobre el esquema
- No se necesitan las features de autenticación de Supabase
- Se prefiere la simplicidad de una base de datos tradicional

Para natales.travel, la simplicidad de PostgreSQL fue perfecta.

### Tailwind v4: menos configuración, más poder

La nueva configuración CSS-based de Tailwind v4 eliminó necesidad de archivos de configuración complejos. Las variables del sistema ahora viven en el CSS puro, lo que simplificó el mantenimiento.

### SEO multilingüe: la importancia del hreflang

Implementar correctamente las etiquetas hreflang impactó directamente en las búsquedas multilingüe. El patrón de URL `/es/slug` y `/en/slug` con canonical tags apropiados mejoró la visibilidad internacional.
