A patient sends a WhatsApp message. They attach a medical exam. The system processes the image, extracts data, calculates pricing, and wraps everything in an automated payment flow. Simple? Behind this sits complex architecture.

## The real challenge

WhatsApp payment flows aren't just buttons. They're complex puzzles with moving pieces:

- Medical document detection from images
- Data extraction with OCR
- Pricing calculation based on exam type
- Identity validation with RUT
- Integration with two different payment providers
- Complex state management in conversations
- PDF receipt generation
- Immediate notification upon payment confirmation

The key isn't in a single component. It's in how everything connects without the user noticing.

## Payment flow architecture

The architecture follows a stage-gate model. First we validate the document is medical. Then we process based on type.

### Stage 1: Document classification

The system receives an image and verifies three things:

- Is it a medical document? (document classifier)
- Is it exam order or results? (key differentiation)
- What specific exam type? (hemogram, lipid, glucose, etc.)

This stage uses a classifier that detects document type before processing. If not medical, the system politely rejects.

```typescript
// Basic classification flow
documentClassifier.classifyDocumentFromStorage()
```

The system learned that confusing hemograms with urine was a critical error. Tests found false positives that led to incorrect flows.

### Stage 2: OCR processing

If the document is valid, the system extracts data. Here there are two paths based on document type:

**For exam orders:**
- Extract patient name
- Validate RUT (required in Chile)
- Identify requested exams
- Calculate price based on catalog

**For exam results:**
- Extract exam type
- Validate normal vs pathological ranges
- Generate automatic interpretation
- Send results with digital signature

OCR uses both OpenAI Vision and local models. The key is in specific prompts for each medical document type.

## Integration with payment providers

Here's the real complexity. Two different payment systems that must behave as one from the user's perspective.

### Mercado Pago: preferred in Chile

Mercado Pago dominates the Chilean market due to WhatsApp Business integration. The flow is:

1. Generate payment preference with amount and description
2. Send payment button with custom URL
3. Wait for payment confirmation webhook
4. Automatically release services upon confirmation

```typescript
// Simplified flow for Mercado Pago
await mercadopago.preferences.create({
  items: [{
    title: exam.name,
    unit_price: exam.price,
    currency_id: 'CLP'
  }]
})
```

### Stripe: for international payments

Stripe requires a different approach. Although it supports WhatsApp, its integration is more complex:

1. Create PaymentIntent with amount and currency
2. Generate Checkout Session URL
3. Redirect user to secure Stripe interface
4. Webhook for confirmation on server

The challenge is maintaining the WhatsApp experience while redirecting to a secure external environment.

## Conversation state management

WhatsApp conversations are naturally stateless. The system must remember context between messages.

We use an in-memory cache with specific patterns:

```typescript
// Pattern: pendingOcr for capturing multiple documents
const contextCache = new Map<string, string>()

function savePendingOcr(userId: string, ocrData: string) {
  contextCache.set(\`pending_ocr_\${userId}\`, ocrData)
}

function getPendingOcr(userId: string): string | undefined {
  return contextCache.get(\`pending_ocr_\${userId}\`)
}
```

This system allows receiving multiple images in sequence and processing them as part of the same order.

## Identity validation

In Chile, RUT is fundamental. We validate identity before any payment:

1. User sends RUT or ID number
2. System validates format and check digit
3. If valid, proceed with payment
4. If invalid, request correct format

Validation happens in the background to not interrupt the conversation.

## Error handling and retries

Payments fail for many reasons: expired card, insufficient funds, network issues.

The system implements:

- Automatic retries with exponential backoff
- Friendly failure notification
- Option to change payment method
- Detailed logging for debugging

We can't leave the user uncertain if something fails.

## User transparency

The key is keeping the user informed without overwhelming them:

- Immediate confirmation upon document receipt
- Real-time processing progress
- Live payment status
- PDF receipt automatically sent

The user always knows what's happening.

## Lessons learned

Building this system taught several important lessons.

First, flow separation is critical. Exam orders and results must process with different logics. An error here causes incorrect flows and lost trust.

Second, payment providers have very different behaviors. Mercado Pago works better for fast payments within Chile. Stripe is necessary for international but breaks the smooth WhatsApp experience.

Third, comprehensive testing is mandatory. We created 13 tests that simulate complete conversations: from first message to payment confirmation. Without them, problems appear in production.

## Future: total interoperability

The next step is connecting this with existing health systems. FHIR, HL7, and Law 21.668 force us to interoperate with clinics, laboratories, and insurance companies.

The current architecture allows this expansion. We can integrate with any system that uses medical standards.

## The real value

What matters isn't that you can charge via WhatsApp. It's that you can reduce friction in the healthcare system.

A patient can order exams from their cellphone. Pay without complications. Receive results without paperwork. All in one natural conversation.

That's the real innovation.
