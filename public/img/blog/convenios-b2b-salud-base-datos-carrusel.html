El Peligro de los Roles Planos
La mayoría de los sistemas legacy manejan convenios añadiendo un rol estático tipo 'CLINIC_ADMIN' a la tabla global de usuarios.
❌ user.role = 'CLINIC_ADMIN' → query.patients({ where: { originClinicId: user.clinicId } })
Si un médico atiende en dos clínicas o el convenio comercial se suspende, tu software no sabrá reaccionar sin edición manual base por base. Estás a una consulta mal filtrada del desastre.

Modelar como Infraestructura
En lugar de roles globales, estructuramos membresías y convenios dinámicos aislados de forma explícita.
✅ table "institution_memberships" { user_id, institution_id, role, status }
Al abstraer la membresía del usuario principal, habilitamos multi-tenancy real. La base de datos restringe físicamente el acceso en base al contexto activo, no a filtros lógicos manuales del desarrollador.

Las 4 Reglas de un Convenio Seguro
1
Aislamiento a nivel de motor: Usar Row-Level Security (RLS) en Postgres para evitar dependencias humanas en el código del controlador.
2
Ciclo de vida dinámico: Si un convenio se congela o revoca comercialmente, el acceso de todos sus miembros debe cancelarse en cascada.
3
Auditoría con contexto: Registrar cada lectura clínica bajo el ID del convenio institucional que autorizó el acceso, no solo el ID del usuario.
